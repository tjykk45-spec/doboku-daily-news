## 気づいたこと（POCで検証すべき点が見えた）

### 1. 「公開日」が本文から取れにくい
- MLIT の報道発表ページは、本文中に日付が明示されていない場合がある（URL階層やパンくずから推定）。
- **コード側の工夫が要る**: URL パターン（例: `houdou202602.html`）や HTML の `<meta>` / `<time>` タグから拾う補助ロジックが必要。
- 取れない場合のフォールバック: 取得日 = `date` として入れ、`lead` に「発表日不明のため取得日で代替」と明記するルールがよい。
→ 2026-04-27: フォールバックルール確定。正本は `design.md` §9 参照。

### 2. 本文が短く、PDF別添が中心の記事がある
- 本ケースもまさにそれ。概要だけでは bullets が薄くなる。
- **方針**: 本文が `N` 文字未満のときは bullets を最大1件に絞り、「詳細は別添PDF参照」を明記する。PDF本文取得は当面**やらない**（コスト・複雑度が跳ねる）。
→ 2026-04-27: 運用ルール確定。正本は `design.md` §9 参照。

### 3. セクション自動判定は意外と揺れる
- 「積算基準改定」は `国交省・規制・政策` と `資材価格・コスト` のどちらにも見える。
- **コード側**: URLのホスト（mlit.go.jp 配下 → 規制・政策寄り）と本文キーワードで重み付け、最後は Gemini のレビュー（[daily-news-review/SKILL.md](./.claude/skills/daily-news-review/SKILL.md)）で `section` を書き換える余地を残すのがよい。

### 4. lead の文字数制限（100文字±20）は現実的
- 今回は103文字で収まった。スキーマのレンジ設定は妥当そう。
- ただし Gemini が「ですます調」を指示通りに守るかは未検証（プロンプトエンジニアリング次第）。

### 5. 「今日の一言」をどう選ぶか未定義
- 朝刊HTMLには「今日の一言」枠があるが、複数記事から1件選ぶルールが書かれていない。
- **提案**:
  - Gemini に「この日の中で最も重要な1本」を選ばせる（別プロンプト）
  - または、`bullets` で使ったキーワードの重複度・ソースの一次性でコード側が選ぶ
- → [design.md](./design.md) 9章「未決事項」に追記すべき。
→ 2026-04-27: 選定ルール確定。正本は `design.md` §9 参照。

### 6. URL ハッシュのキー設計
- URL 全体だとクエリ・トラッキング付きで揺らぐ可能性。
- **方針**: `url_hash` はドメイン除いたパス末尾（例: `kanbo08_hh_001304`）で十分そう。今回はそれで作った。

### 追加で見えた気づき（スキーマ頑健性のストレステスト）

#### 7. スコープ境界（土木 vs 建築）
- 2本目は「公共**建築**工事」の話で、**本朝刊のスコープ（土木中心）からは外れる**。
- しかし管理職にとって費用構造の参考値としては有用。
- **提案**: スキーマに `is_in_scope: true|false` フィールドを追加し、`false` でも朝刊には載せつつ「スコープ外参考」バッジを付ける。
- または、「スコープ判定」を**朝刊に載せる前のフィルタ**として分離する（これが [design.md](./design.md) の「毎日量産しない」ゲートと同じ場所に入ると見通しがよい）。

#### 8. 「発表日」と「内容の日付」のずれ
- 3本目は発表日2026-04-03だが、提言内容そのものは中期的な話（施策化は未来）。
- 読者の「いつ動くか」という関心からは、**発表日だけでは不十分**で、「施策化の見通し」情報が欲しくなる。
- **提案**: bullets の1つに「施策化までの見込み」または「次会議体の立ち上がり」を書くルールにする。

#### 9. 文字化け懸念（丸数字）
- 3本目本文に丸数字（①②③）が含まれており、取得時 `[1][2][3]` にフラット化された。
- **提案**: HTMLスクレイピング後に丸数字や特殊記号の正規化を入れるか、AIプロンプトで「原文記号を保持」と明示する。
- 今回は lead・bullets で丸数字に触れずに書き直したので実害なし（むしろ読みやすくなった）。

#### 10. 「提言段階」vs「施策」の粒度
- 3本目は**提言段階**で、まだ制度化されていない。
- 2本目は**確定した改定**で、4月以降即適用。
- 同じ `国交省・規制・政策` でも粒度が違う。bullets で「確定／提言」の扱いを明示したほうがよい。
- **提案**: タグに `#確定` `#提言` を追加するか、`section` を2段階（大セクション + サブタグ）にする案もある。

#### 11. title が80文字を超えやすい
- 3本目の生成 title は「国交省「今後の建設業政策のあり方に関する勉強会」とりまとめ公表 月給制転換やDX・経営評価見直しを提言」＝約50文字で収まったが、**原文タイトルが長いケースでは80文字超えのリスク**あり。
- **提案**: スキーマの title 上限を90文字に緩和するか、長文タイトルは「主語：提言内容」の形で機械的に短縮するルールを明記する。

## 「今日の一言」選定ルール草案（2026-04-27）

### 基本方針
その日の記事の中から**管理職が最も意思決定・行動に直結する1本**を選ぶ。
「面白い」ではなく「明日の仕事に影響する」を選定軸とする。

### 選定の優先順位（スコア式）

以下の観点を順番に評価し、最も高い記事を選ぶ。

1. **即時性**（最優先）
   - `maturity: 確定` の記事を `maturity: 提言` より優先する
   - 施行日・適用日が近い（3ヶ月以内）記事はボーナス

2. **スコープ一致**
   - `is_in_scope: true` の記事を優先する
   - `is_in_scope: false` は他に候補がない場合のみ選ぶ

3. **ソースの一次性**
   - 国交省・官公庁の発表 ＞ 業界紙報道 ＞ その他

4. **影響範囲**
   - 全国・業界横断の話題 ＞ 地域・個社の話題

### 実装方式（2択）

**Option A：Geminiに選ばせる（推奨）**
その日の全記事のJSONを渡し、以下のプロンプトで1本選ばせる：
> 「以下の記事の中から、土木建設業の管理職が明日の意思決定に最も直結する1本を選び、
> その記事のURLと50文字以内の選定理由を返してください。」

**Option B：コードで選ぶ（シンプル）**
上記の優先順位をスコア化してコード側で機械的に選ぶ。
- `maturity=確定` → +3点
- `is_in_scope=true` → +2点  
- ソースが国交省 → +2点
- 全国案件 → +1点

### フォールバック
- その日の記事が1本のみ → 自動的にその記事を選ぶ
- 記事が0本 → 「今日の一言」欄は非表示にする（エラー扱いにしない）

### 結論
当面はOption Aで実装し、選定結果がおかしいと感じたらスコアルールを調整する。
Geminiへの追加プロンプトは `daily-news-summarize` スキルとは別の
`daily-news-pick` スキルとして独立させる。
